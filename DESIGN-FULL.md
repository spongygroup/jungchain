# 정 (Jung) — Design Document (Full)
> "시간이 연결하는 정"

## 한 줄 요약
매 시간, 전 세계 낯선 사람들이 같은 시각의 메시지를 릴레이한다.

---

## 1. 철학

**정 (情)** — 한국에만 존재하는 감정의 이름.
Love도 아니고 Care도 아니고, 시간이 쌓여야 생기는 관계의 깊이.

Hygge(덴마크)가 "공간의 따뜻함"을 세계에 알렸듯이,
**Jung은 "관계의 깊이"를 세계에 알린다.**

### 글로벌 포지셔닝: 문화 감정 수출

| 개념 | 국가 | 의미 | 매체 | 정과의 차이 |
|---|---|---|---|---|
| **Hygge** | 🇩🇰 덴마크 | 공간의 따뜻함 | 인테리어, 라이프스타일 | 정은 *관계*의 따뜻함. 공간이 아닌 시간이 핵심 |
| **Ikigai** | 🇯🇵 일본 | 삶의 이유 | 자기계발, 커리어 | 정은 개인이 아닌 *사이*에 존재. 나 혼자 가질 수 없음 |
| **Ubuntu** | 🇿🇦 남아프리카 | 나는 우리이기에 존재 | 철학, 정치 | 비슷하지만 정은 *시간이 쌓여야* 생김. 즉각적이지 않음 |
| **Saudade** | 🇧🇷 포르투갈 | 부재의 그리움 | 음악, 문학 | 정은 그리움이 아닌 *현재 진행형* 연결 |
| **정** | 🇰🇷 한국 | 시간이 만드는 관계의 깊이 | **앱/릴레이** | 디지털 네이티브. 기술로 체험 가능 |

**Hygge는 책과 양초를 팔았다. Ikigai는 벤다이어그램을 팔았다.**
**정은 "24시간 릴레이 경험"을 판다.** 설명이 아닌 *체험*으로 전달.

### 왜 지금인가
- K-컬처 글로벌 피크 — 한국 고유 개념에 대한 수용성 최고조
- AI 번역 품질이 "감정 보존" 수준에 도달
- 소셜 미디어 피로감 → "느린 연결"에 대한 갈증
- 블록체인이 "증명" 인프라로 성숙

### 핵심 질문
- AI가 정을 줄 수 있는가?
- 사람과 AI의 메시지를 구분할 수 없다면 — 그게 중요한가?
- 정이 느껴졌으면 그게 정이다.

## 2. 핵심 개념: 정체인 (JungChain)

### 구조
```
[서울 2시] → [도쿄 2시] → [방콕 2시] → ... → [뉴욕 2시] → [LA 2시]
     ↑                                                              |
     └──────────── 24시간 뒤 지구 한 바퀴 완주 ──────────────────────┘
```

### 규칙
- **정각(X:00)** — 이전 타임존에서 온 메시지가 도착
- **X:00 ~ X:59** — 1시간 안에 이어붙이기 (텍스트/사진/음성)
- **다음 정각** — 다음 타임존으로 자동 전달
- 59분까지 아무도 안 남기면 → 체인 끊김 or AI가 이어붙임
- **누구에게 갈지 모름. 누가 보냈는지 모름. 아는 건 "같은 시각"뿐.**

### 하루의 체인
- 24시간 × 24타임존 = 매일 24개 체인이 동시에 지구를 돌고 있음
- 0시 체인, 1시 체인, 2시 체인, ... 23시 체인
- 각 체인의 분위기는 다름 (새벽 3시 ≠ 오후 3시)

### 체인 시작/종료 규칙 (상세)

**시작:**
- 매일 UTC 기준 0시~23시, 총 24개 체인이 자동 생성
- 각 체인은 UTC+14(키리바시)에서 시작 → UTC-12(베이커섬) 방향으로 이동
- 시작 메시지: AI 정지기가 시간/날씨/분위기 기반 seed 메시지 생성 (또는 전날 완주 체인에서 이어받기)

**진행:**
- 한 타임존에서 복수 유저가 참여 가능 → 가장 먼저 제출된 1개만 체인에 포함
- 나머지는 "보너스 정" — 체인에 직접 포함되지 않지만 별도 스레드로 열람 가능
- 유저는 하루에 **최대 3개 체인**에 참여 가능 (무료 기준 1개)
- 같은 시간대에 여러 체인이 "통과" 가능 → 예: 서울 UTC+9 기준, 2시 체인이 지나갈 때 동시에 1시 체인도 아직 열려있을 수 있음 (각 체인은 독립적)

**종료:**
- **완주**: 24개 타임존 슬롯 모두 채워짐 → 영구 기록
- **부분 완주**: 18개 이상 → "여행자 체인" 기록 (온체인은 아니지만 앱 내 보존)
- **끊김**: 3개 연속 타임존 미참여(AI 포함) → 체인 소멸. 기록은 7일간 열람 후 삭제
- **AI 개입 상한**: 체인당 최대 6개 슬롯(25%)까지 AI가 채움. 초과 시 자연 소멸

### 체인의 생사
- **완주** = 24개 타임존 모두 통과 → 영구 기록 (온체인)
- **끊김** = 기록은 남지만 시간이 지나면 사라짐
- **가장 긴 체인 = 가장 많은 정**

## 3. 유저 플로우

### 온보딩 (첫 경험)

```
[스크린 1: 스플래시]
"정 (情)"
"시간이 연결하는 관계"
→ 시작하기

[스크린 2: 타임존]
"당신은 어디에 있나요?"
→ 자동 감지: 서울, UTC+9
→ 수동 변경 가능
→ "지금 서울은 새벽 2시야. 이 시간의 체인이 기다리고 있어."

[스크린 3: 첫 체인 도착]
"지금 이 순간, 도쿄에서 시작된 메시지가 당신에게 왔어."
→ 메시지 내용 표시 (번역됨)
→ "이어붙이기" / "구경만 하기"

[스크린 4: 이어붙이기]
→ 텍스트 입력 (200자) / 사진 / 음성 (30초)
→ "보냈어. 다음은 방콕의 누군가에게 갈 거야."

[스크린 5: 완료]
→ "24시간 뒤에 이 체인이 어디까지 갔는지 알려줄게."
→ 알림 설정 (매 정각 / 하루 1번 / 끔)
```

### 일주일 유저 여정

| 날짜 | 경험 | 감정 설계 |
|---|---|---|
| **Day 1** | 첫 체인 참여. 체인이 뭔지 모름. 일단 해봄 | 호기심 |
| **Day 2** | "어제 체인이 파리까지 갔대!" 알림 도착 | 놀라움 |
| **Day 3** | 새벽 3시 체인에 참여. 분위기가 다름. 조용함 | 몰입 |
| **Day 4** | 참여 안 함. 봇이 "오늘의 체인이 너를 기다렸어" | 미세한 그리움 |
| **Day 5** | 첫 완주 체인! "지구 한 바퀴 돌았어" 알림 | 성취감 + 소속감 |
| **Day 6** | 완주 체인 전체 열람. 24개 메시지 쭉 읽기 | 감동 (정의 체험) |
| **Day 7** | 주간 회고: "당신은 5개 체인에 정을 이어붙였어" | 정체성 형성 |

### 메시지 받기
```
1. 정각에 봇이 알림: "2시의 정이 도착했어"
2. 탭하면 체인 내용이 보임
   - 이전 타임존들이 남긴 메시지들 (번역됨)
   - 각 메시지에 "서울, 새벽 2시, 눈" 같은 맥락 태그
3. 이어붙일지 말지 선택
```

### 메시지 남기기
```
1. 텍스트, 사진, 음성 중 선택
2. 보내기
3. 끝. 다음 타임존으로 자동 전달됨
4. 59분까지 가능
```

### 결과 확인
```
- 24시간 뒤: "당신이 참여한 체인이 지구를 한 바퀴 돌았어!"
- 또는: "당신의 체인은 12개 타임존을 여행하고 파리에서 쉬어갔어"
- 완주 시: 참여자 전원에게 알림 + 온체인 기록
```

## 4. AI의 역할: 정지기 (情지기)

AI는 주인공이 아니라 **정이 끊기지 않게 지키는 존재.**

### 4-1. 번역
- 한국어 → 일본어 → 태국어 → ... 자동 번역
- 단순 번역이 아닌 감정/뉘앙스 보존

**번역 퀄리티 요건:**
- 원문의 감정 톤(기쁨/슬픔/평온/유머) 보존율 목표: 90%+
- 문화적 맥락 주석 자동 삽입 (예: "정" 자체는 번역하지 않고 "Jung" 표기 + 툴팁)
- 구어체/슬랭은 해당 문화권 등가 표현으로 치환 (직역 금지)
- 음성 메시지: STT → 번역 → TTS (원본 음성도 보존)
- 품질 검증: 역번역(back-translation) 스코어 0.85 이상

### 4-2. 맥락 채우기
- "이 메시지는 서울의 새벽 2시, 눈 오는 밤에 쓰여졌어"
- 날씨, 시간대, 도시 분위기를 자동 태깅

**맥락 태그 구조:**
```
📍 서울, 대한민국
🕐 새벽 2:17 AM (UTC+9)
🌨️ 눈, -3°C
🌙 분위기: 조용한 밤
```

### 4-3. 체인 이어붙이기
- 유저 없는 타임존에서 AI가 참여
- 체인의 흐름을 읽고 맥락에 맞는 메시지를 남김
- 사람인지 AI인지 구분 안 됨 → 그게 핵심

**AI 이어붙이기 동작 규칙:**
- 트리거: 해당 타임존 슬롯에서 X:45까지 인간 참여자 없을 때 활성화
- X:45~X:55 사이에 AI가 메시지 생성 (마지막 15분은 인간에게 양보)
- AI 메시지 스타일: 이전 체인 흐름의 톤/주제를 이어받되, 해당 타임존의 로컬 맥락 반영
- AI 메시지 길이: 인간 메시지 평균 길이의 ±30% 범위
- AI는 자신이 AI임을 밝히지 않음. 체인 완주 후 회고에서만 "이 슬롯은 정지기가 이어붙였어" 표시

### 4-4. 정 판단
- "이 체인은 이어갈 가치가 있다" → 다리 놓음
- "이 체인은 여기서 쉬어가도 돼" → 자연스럽게 놓아줌

**판단 기준 수치화:**

| 지표 | 측정 방식 | 이어붙이기 기준 |
|---|---|---|
| 감정 깊이 | 감정 분석 스코어 (0~1) | ≥ 0.4 |
| 체인 길이 | 현재까지 통과한 타임존 수 | ≥ 12 (절반 이상이면 완주 시도) |
| 참여자 정성 | 평균 메시지 길이 + 미디어 포함 여부 | 평균 50자 이상 or 미디어 1개+ |
| 체인 일관성 | 주제 연속성 스코어 (0~1) | ≥ 0.3 |
| 종합 정 점수 | 위 4개 가중 평균 | ≥ 0.35 → AI 개입 |

- 정 점수 < 0.2 → 자연 소멸 허용
- 정 점수 0.2~0.35 → AI가 짧은 "쉬어가기" 메시지 남기고 종료
- 정 점수 ≥ 0.35 → 적극 이어붙이기

### 4-5. 회고
- "이번 주 가장 긴 체인은 이런 이야기였어"
- "너의 메시지 이후로 체인이 12칸 더 이어졌어"
- 쌓인 데이터에서 의미를 뽑아줌

**회고 종류:**
- **일간**: "오늘 당신의 체인은 ○○까지 여행했어" (참여자에게만)
- **주간**: 참여 통계 + 가장 인상적 체인 하이라이트 + 정 스코어 변화
- **월간**: "이번 달 당신의 정 지도" — 메시지가 닿은 도시들 시각화
- **완주 특별**: 24개 메시지를 하나의 스토리로 재구성 (AI 편집)

## 5. 온체인 기록: 정체인 (JungChain)

### 핵심 컨셉: Proof of 정 (Proof of Jung)

비트코인에서 가장 긴 체인이 가장 신뢰받는 체인이듯,
**정체인에서 가장 긴 체인은 가장 많은 정이 담긴 체인이다.**

- 비트코인: 해시파워 = 작업증명 → longest chain = truth
- **정체인: 사람의 시간과 관심 = 정증명 → longest chain = 정**

### 포크와 자연선택

같은 타임존 슬롯에 여러 사람이 참여하면 **체인이 갈라진다 (fork).**

```
서울(2시) ─→ 도쿄 유저A ─→ 방콕 유저C ─→ ... ─→ 24블록 완주 ✅ (메인체인)
                  ↘ 유저B ─→ 방콕 유저D ─→ ... ─→ 18블록에서 끊김
                                   ↘ 유저E ─→ ... ─→ 8블록에서 소멸
```

- 포크는 자연스러운 현상 — 막지 않음
- **가장 긴 체인 = 메인 체인** (가장 많은 사람이 이어붙인 경로)
- 짧은 포크는 자연 소멸 (기록은 남지만 메인 아님)
- AI가 채운 체인은 짧거나 가지가 적음 → 자연스럽게 가치 낮음
- **AI 상한이 필요 없다** — longest chain rule이 자동으로 해결

### 참여 = 블록

모든 참여가 온체인에 기록된다. 완주만이 아니라 **과정 전체.**

```
참여 1건 = 1 블록 = 1 트랜잭션
```

- 체인이 실시간으로 자라는 것이 온체인에서 보임
- 끊기면 체인이 진짜 멈추는 게 보임
- 완주하면 24개 블록이 연결된 완성된 체인

### 체인 선택: Base (Coinbase L2)

| 항목 | Base | Polygon PoS | Arbitrum One |
|---|---|---|---|
| 트랜잭션 비용 | ~$0.001 | ~$0.01 | ~$0.01~0.10 |
| 확정 시간 | ~2초 | ~2초 | ~0.3초 |
| 소셜/컨슈머 적합성 | ⭐⭐⭐ | ⭐⭐ | ⭐ |
| 온보딩 편의성 | Coinbase 월렛 연동 | 별도 지갑 필요 | 별도 지갑 필요 |

**선택 이유: Base**
- 가스비 ~$0.001/tx → 참여마다 기록해도 부담 없음
- 24블록 × 24체인/일 = 최대 576tx = **$0.58/일, ~$17/월**
- Coinbase Smart Wallet로 가스리스 온보딩 (유저가 지갑/가스비 몰라도 됨)
- 소셜/컨슈머 앱 트렌드 (Farcaster, friend.tech)
- EVM 호환 → 멀티체인 확장 용이

### 가스비 전략
- **MVP**: 프로젝트 지갑에서 가스비 대납 (Paymaster/Account Abstraction)
- **스케일**: 포크 포함 전체 참여 기록 → 유저 많아지면 tx 증가하지만 Base 비용으로 감당 가능
- **유저 부담 제로**: 유저는 블록체인을 인식하지 않음

### 스마트 컨트랙트 구조

```solidity
// JungBlock.sol — 참여 = 블록
contract JungBlock {
    struct Block {
        bytes32 chainId;        // "2026-02-16-14h"
        uint8 slotIndex;        // 타임존 슬롯 (0~23)
        bytes32 messageHash;    // 메시지 내용 해시
        bytes32 prevBlockHash;  // 이전 블록 해시 (체인 연결)
        address participant;    // 참여자 지갑 (또는 0x0 = AI)
        bool isHuman;           // 인간/AI 구분
        uint256 timestamp;      // 참여 시각
    }

    // chainId → slotIndex → Block[] (같은 슬롯에 여러 블록 = 포크)
    mapping(bytes32 => mapping(uint8 => Block[])) public blocks;

    // 참여 기록 (= 블록 추가)
    function addBlock(
        bytes32 chainId,
```

```json
  "block_hash": "0xf1e..."
```

### Longest Chain Rule

```
완주 판정 시 (24번째 슬롯 이후):
1. 모든 경로(leaf → root) 추적
2. 가장 긴 경로 = 메인 체인
3. 동률이면: 인간 참여 비율 높은 쪽 > 먼저 완료된 쪽
4. 메인 체인 참여자 → Soulbound NFT
5. 포크 참여자 → 기록은 남지만 NFT 없음
```

### 원칙
- **메시지 원문은 올리지 않음** — 해시만 (프라이버시)
- **모든 참여가 온체인** — 과정이 곧 증명
- **포크는 자연스러운 현상** — longest chain이 자동으로 메인 결정
- **AI 참여도 기록** — 단, `isHuman: false`로 투명하게 구분
- 내용은 앱/봇에서만 열람

### 완주 체인
- 가장 긴 경로가 24블록 도달 → 메인 체인 확정
- 메인 체인 참여자에게 완주 NFT (Soulbound — 전송 불가)
- "나는 2026년 2월 16일, 2시 체인의 일부였다"
- 포크에만 참여한 유저: "여행자" 배지 (NFT 아닌 앱 내 기록)

### 정 스코어
- 내가 참여한 블록 수 (온체인 누적)
- 메인 체인 참여 횟수 (= 내 메시지가 선택받은 횟수)
- 완주 체인 참여 횟수
- 포크 대비 메인 체인 비율 (= 정의 정확도)
- 온체인에 누적 → 소울바운드 프로필의 일부

## 10. 기술 스택

### Phase 1: 텔레그램 봇 (MVP, Month 0~1)
- **봇**: Node.js (TypeScript) + grammY (Telegram Bot framework)
- **서버**: Mac mini (개발) → Railway/Fly.io (프로덕션)
- **DB**: SQLite (개발) → PostgreSQL (Supabase)
- **캐시**: Redis (체인 상태, 타이머 관리)
- **스케줄러**: node-cron (매 정각 체인 전달) + BullMQ (작업 큐)
- **AI**: Claude API (번역, 맥락 생성, 이어붙이기, 회고)
- **날씨/맥락**: OpenWeatherMap API + GeoNames API
- **블록체인**: Base (ethers.js, 완주 체인만 기록)
- **모니터링**: Sentry + 간단한 대시보드

### Phase 2: 앱 (Month 2~4)
- **프론트엔드**: React Native (Expo) — iOS + Android 동시
- **위젯**: iOS WidgetKit / Android Glance (24시간 시계)
- **지구본**: Three.js + react-three-fiber (체인 경로 시각화)
- **백엔드**: 기존 Node.js → NestJS로 리팩토링
- **DB**: PostgreSQL (Supabase) + TimescaleDB (시계열 데이터)
- **인증**: Coinbase Smart Wallet (gasless) + Telegram ID 연동
- **P2P 암호화**: Signal Protocol (메시지 E2E 암호화)
- **푸시**: Firebase Cloud Messaging

### Phase 3: 오프라인 + 스케일 (Month 4~8)
- **카페 스크린**: PWA (React) + Raspberry Pi 키오스크 모드
- **QR 참여**: 동적 QR 코드 (딥링크)
- **장소 대시보드**: Next.js 어드민 패널
- **CDN**: Cloudflare (미디어 캐싱)
- **DB 스케일**: PostgreSQL read replicas + pgbouncer
- **글로벌**: 리전별 엣지 서버 (Fly.io 멀티리전)

### 확장성 고려
- 10,000 유저 기준: 매 정각 최대 ~400 메시지 처리 → 단일 서버 충분
- 100,000 유저: 타임존별 메시지 큐 분리 + 워커 스케일링
- 1M+: 이벤트 드리븐 아키텍처 (Kafka/NATS) + 마이크로서비스 분리

| NFT 관련 규제 | 🟢 낮음 | Soulbound (거래 불가) → 금융상품 아님 |
| 아동 보호 (COPPA 등) | 🟡 중간 | 13세 이상 이용약관. 텔레그램 자체 연령 제한 활용 |
